# セトリ予想ビンゴ

セットリスト予想ビンゴゲーム。複数の「大会」をプルダウンで選び、大会ごとに予想・ベット・自動採点・結果・流行が楽しめる。

ビルド不要の素のHTML/CSS/JavaScript(単一ファイル)。GitHub Pagesでそのまま公開できる。バックエンドはSupabase(無料枠)。モバイルファースト設計(常に375px幅で表示、PCでは両脇が背景になる)。

## タブ構成

- **予想**: プロデューサー名刺・ビンゴ・ベットの入力
- **流行**: みんなの予想傾向(確定済み参加者の集計)
- **結果**: 順位・スコア(正解セトリが登録されると自動的に表示される。それまでは非表示)
- **管理**: 大会作成者のみ表示

## できること

- **チュートリアル**: 初回ログイン時に6ステップの使い方ガイドを自動表示。ヘッダーの「使い方」からいつでも見返せる
- **プロデューサー名刺**: プロデューサー名・ひとこと・担当アイドル(アイドル名(C.V.声優名)形式)を設定
- **セトリ予想**: 3×3マスにセトリ予想を入力。左上=1曲目(OPENING)・右下=ラスト曲(ENCORE)は曲順まで的中判定、それ以外の7マスは「セトリ全体のどこかに入っていれば的中」
- **夢に賭ける**: 12,300ジュエルを、倍率でグループ化した3択(マスター/プロ/レギュラー)でベット。グループへの入力は、そのグループ内のライン数で自動的に均等分配される
- **保存 = 確定**: 画面右下のフロートボタン「保存する」を押すと、その場でDBに保存され、確定済み(結果反映・流行反映の対象)になる。締切までは何度でも保存し直せる。締切、または正解セトリ登録後は自動的にロックされる
- **自動採点(一括)**: 管理者が正解セトリを登録すると、確定済みの参加者**全員がその場で自動的に採点**される。各参加者がログインし直す必要はない。締切前でも、正解セトリが登録された時点で予想は締め切られ、結果表示に切り替わる
- **流行タブ**: 確定済み参加者全員の予想を集計した「みんなの予想傾向」(担当アイドル人気ベスト3・1曲目予想・ラスト曲予想・全体人気曲)。数時間おきに自動更新、管理者は手動でも即時集計できる
- **結果タブ**: 「トップ10」「自分の周辺(自分の順位の前後5人)」を切り替えて表示。多人数の参加を想定した設計
- **SNS共有**: セトリ予想ビンゴの図・ベット状況・担当アイドル・(確定後は)最終ジュエルとガシャ換算を1枚の縦長JPG画像にして生成し、保存(対応端末では写真に保存される)またはXへ共有
- **管理タブ**: 大会作成者だけがアクセス可能。正解セトリのCSV登録(上書き対応)、候補曲リスト/担当アイドル一覧の切り替え、流行の手動集計、締切変更、確定済み全員の再採点、大会削除ができる

## 盤面構成

3×3マス(FREEマスなし、9マス全部に予想を入力)。左上=1曲目(OPENING)、右下=ラスト曲(ENCORE)で、この2マスだけ曲順まで的中判定。それ以外の7マスは、セトリ全体のどこかに入っていれば的中。

```
0 1 2
3 4 5
6 7 8
```

## ベットとライン別の払戻倍率(BINGO成立時)

8ライン(横3・縦3・斜め2)を、倍率で3グループに分けてベットする。

| グループ | 倍率 | 対象ライン | 理由 |
|---|---|---|---|
| マスター | ×765 | 斜め↘(0,4,8) | OPENING・ENCORE両方を含む(最難。滅多に当たらない代わりに大化けするロマン枠) |
| プロ | ×8 | 横1・横3・縦1・縦3(4本) | OPENING/ENCOREのどちらか片方を含む |
| レギュラー | ×4 | 横2・縦2・斜め↙(3本) | 特別マスを含まない通常ライン |

グループへの入力額は、そのグループ内のライン数で均等に自動分配される(例: プロに4,100ジュエル→4本のラインに1,025ずつ)。初期ベットは各グループ3,000ジュエルずつ(合計9,000。残り3,300は未配分)。

担当アイドルの声優(CV)が、BINGO成立ラインの曲の歌唱者と一致していると、該当マス1つにつきさらに×2(累積)。OPENING・ENCORE的中はそれぞれ最終合計に×2(両方的中で×4)。

## 候補曲・担当アイドルは、どちらも大会ごとに切り替え可能(静的ファイル)

候補曲リストも担当アイドル一覧も、複数のJSONファイルを用意して**大会ごとにどれを使うか選べる**(大会作成時・管理タブどちらからも変更可能)。利用可能なファイルの一覧は、それぞれ `songs-manifest.json` / `casts-manifest.json` に登録する。

```json
// songs-data.json の例(候補曲リストの1つ)
["モーニング・プリズム", "リハーサル・トワイライト", "..."]

// songs-manifest.json の例(利用可能な候補曲JSON一覧)
[
  { "id": "gakumas", "label": "学園アイドルマスター(テスト)", "file": "songs-data.json" },
  { "id": "gaku", "label": "GAKU", "file": "songs-data-gaku.json" }
]

// casts-data.json の例(担当アイドル一覧の1つ。アイドル名+声優CV)
[{"idol": "花海咲季", "cv": "石見舞菜香"}, {"idol": "月村手毬", "cv": "前田佳織里"}]

// casts-manifest.json の例(利用可能な担当アイドルJSON一覧)
[
  { "id": "gakumas", "label": "学園アイドルマスター(テスト)", "file": "casts-data.json" },
  { "id": "another", "label": "別のアイドルグループ", "file": "casts-data-2.json" }
]
```

新しい候補曲リスト/担当アイドル一覧を追加したい場合は、①新しいJSONファイルをこのリポジトリに追加し、②対応するマニフェストにその情報(id/label/file)を追記してpushする。それだけで、大会作成時の選択肢・管理タブの選択肢に自動的に出てくる。

正解セトリ(実際に演奏された曲+歌唱者)は、これとは別に大会ごとにCSVで管理タブから登録する。歌唱者は声優(CV)名で記録し、担当アイドル選択もCV名で照合する。CSV形式は「曲名,歌唱者」。最初のカンマだけで区切るので、歌唱者が複数いる場合はそのままカンマ区切りでOK(例: `SUPREMACY,長月あおい, 小鹿なお, 飯田ヒカル`)。「ユニット名(メンバー1, メンバー2)」形式や「全員」(登録済み声優全員として展開)にも対応。既に登録済みの内容がある場合は、上書き確認のダイアログが出る。

## 1. Supabaseプロジェクトを作る

1. https://supabase.com にログイン(なければアカウント作成)
2. 「New project」でプロジェクトを作成(リージョンは Tokyo (ap-northeast-1) を推奨)
3. 左メニュー **SQL Editor** を開き、`supabase-schema.sql` の中身を全部貼り付けて実行
   - `create extension if not exists pg_cron;` の行でエラーが出た場合(プランでpg_cronが使えない場合)は、その行と直後の `select cron.schedule(...)` をスキップして残りだけ実行してよい。その場合、流行の集計は管理タブの「今すぐ集計する」ボタンでの手動運用になる
   - 既存プロジェクトに反映するだけの場合は、`alter table ... add column if not exists ...` や `drop policy if exists ... / create policy ...` の行だけ実行すれば、既存データを消さずに反映できる
4. 左メニュー **Authentication > Providers > Email** で **「Confirm email」をOFF**
5. 左メニュー **Project Settings > API** で **Project URL** と **anon public key** を確認

## 2. Supabaseの接続情報を書き込む

`index.html` 内、`<script type="module">` の先頭付近にある以下を書き換える。

```js
const SUPABASE_URL = "あなたのプロジェクトURL";
const SUPABASE_ANON_KEY = "あなたのanon key";
const FAKE_EMAIL_DOMAIN = "gakumas-bingo.local"; // ログインIDを疑似メール化する際のドメイン
const SUPER_ADMIN_LOGIN_ID = "gakuen"; // 大会を作成できる管理者のログインID
```

> anon keyはpublicで問題ない値(Supabase設計上、フロントエンドに埋め込む前提のキー)。データの保護はRLS(Row Level Security)が担っている。

**重要**: `supabase-schema.sql`内の「大会の作成」ポリシー(`events insert admin only`)は、`SUPER_ADMIN_LOGIN_ID`に対応する固定メールアドレス(`ログインID@FAKE_EMAIL_DOMAIN`、例: `gakuen@gakumas-bingo.local`)で実際に照合している。`SUPER_ADMIN_LOGIN_ID`や`FAKE_EMAIL_DOMAIN`を変更した場合は、SQL側の該当箇所(`auth.email() = '...'`)も必ず合わせて変更すること。ここがズレると「アプリ上はボタンが隠れているだけで、実際は誰でも大会を作成できてしまう」状態になる。

すでにpush済みのリポジトリなら、`git add . && git commit -m "update" && git push` でOK。

## 4. GitHub Pagesを有効化する

Settings > Pages で Source を `Deploy from a branch` / ブランチ `main` / フォルダ `/(root)` にする。

`index.html`は`songs-data.json`・`casts-data.json`などを相対パス(`./songs-data.json`など)で読み込むので、必ずリポジトリのRoot直下に全ファイルをまとめて置くこと。

## 大会の使い方

1. `SUPER_ADMIN_LOGIN_ID` のIDでログインし、「＋ 新しい大会を作る」で大会名・開催日・締切・候補曲リスト・担当アイドル一覧を選んで作成
2. 参加者は予想がまとまったら画面右下の「保存する」を押す(締切前ならこれが確定を兼ねる。締切までは何度でも保存し直せる)
3. ライブ終了後、大会作成者は「管理」タブから正解セトリのCSVを登録する。登録すると、確定済みの参加者が自動で一括採点され、「結果」タブが全員に表示されるようになる。予想タブも「プロデューサー名刺・ビンゴ・列のベット・スコア・SNS共有」だけのシンプル表示に切り替わる
4. 「流行」タブでみんなの予想傾向を確認、「結果」タブで成績(トップ10/自分の周辺)を確認できる

## セキュリティについて

このアプリはサーバーを持たない構成のため、データの保護はすべてSupabaseのRLS(Row Level Security)ポリシーに依存している。アプリのHTML/JSはブラウザの開発者ツールで誰でも書き換え・直接API呼び出しができる前提で設計してあり、「見た目のボタンを隠す」ことと「実際にDBへの操作を制限する」ことは別物として扱っている。

- 大会の作成・削除・締切変更・正解セトリ登録は、実際の管理者アカウント(`events.created_by`、および`events insert admin only`ポリシーでの`auth.email()`照合)にのみ許可
- 各参加者の予想(`game_saves`)・結果(`results`)は本人のみ書き込み可能。大会管理者は自分が作成した大会分に限り、確定済みデータの読み取り・一括採点書き込みが可能
- SQLインジェクションについては、Supabase JS SDK(`.select()`/`.insert()`など)経由のパラメータ化されたAPI呼び出しのみを使用しており、生SQLの文字列組み立ては行っていない

なお、「Confirm email」をOFFにしている都合上、アカウントの大量作成(スパム)自体は現状防げていない。実害は小さいと考えているが、気になる場合はSupabase側でCAPTCHA(hCaptcha/Turnstile)を有効化する設定を検討してほしい。

## ファイル構成

```
index.html                画面・見た目・ロジックすべてを含む単一ファイル
                           (★中盤のSupabase接続情報とSUPER_ADMIN_LOGIN_IDだけ書き換える)
songs-data.json            候補曲リストの1つ。複数用意できる
songs-manifest.json        利用可能な候補曲一覧JSONの目次(大会ごとにここから選ぶ)
casts-data.json            担当アイドル一覧の1つ(アイドル名+声優CV)。複数用意できる
casts-manifest.json        利用可能な担当アイドル一覧JSONの目次(大会ごとにここから選ぶ)
supabase-schema.sql        Supabaseに流し込むテーブル定義・RLS設定・トレンド集計関数
```

## 補足・今後の拡張余地

- 大会の管理者は`SUPER_ADMIN_LOGIN_ID`の1アカウントのみ(複数管理者には未対応)
- 一括採点・流行集計は、管理者の操作をきっかけに動く設計(数千〜数万人規模のアクセスを想定した完全サーバーレス分散処理ではなく、利便性を優先した設計)
- pg_cronがSupabaseのプランで使えない場合は、流行集計は管理タブの手動ボタンで運用する
- 「保存」＝「確定」に統一しているため、下書き保存(9マス未入力・ベット未配分の状態での保存)はできない仕様
